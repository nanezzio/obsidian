### Виды тестирования ПО

Дан код:
```
void compare(int a, int b)
{
	if (a > b)
		std::cout << a << " больше " << b << '\n';
	else if (a < b)
		std::cout << a << " меньше " << b << '\n';
	else
		std::cout << a << " ровно " << b << '\n';
}
```

Как тестировать?
Чтобы протестировать работу этого кода, необязательно сравнивать все пары чисел типа int. Для этого можно протестировать всего 3 пары чисел, где:
- одно больше другого
- они равны
- одно меньше другого

**Важно**
Тесты должны быть всесторонними, т.е. тесты должны учитывать разные варианты входных данных, при этом функция должна работать корректно во всех случаях. Проверьте крайние случаи, такие как передача нулевых значений или пустых строк. Убедитесь, что функция также обрабатывает их правильно

Основные виды тестирования:
- Модульное тестирование (unit) - исследуют, как работают отдельные функции или методы
Пример: Есть класс, который представляет математические операции, и вам нужно протестировать метод сложения чисел. Модульный тест проверит корректность метода сложения чисел и его работу внутри класса

- Сквозное тестирование (End to end) - тестируют, как различные части программы взаимодействуют друг с другом в реальных условиях
Пример: При разработке приложения для банка тестировщик воссоздает путь, который пользователь проходит, начиная со входа клиента в приложение, просмотра баланса, перевода средств, и заканчивая получением подтверждения операции
Проверка сценариев:
- Простые - тестирование авторизации
- Сложные - тестирование проведения долгих расчетов

- Интегральное тестирование (Integration) - Проверяют, как компоненты программы интегрируются друг с другом
Пример: Есть веб-приложение, которое должно взаимодействовать с базой данных для хранения и извлечения информации. Интеграционное тестирование позволит убедиться, что приложение правильно взаимодействует с базой данных, сохраняя и извлекая данные без ошибок

- Тестирование производительности (Perfomance) - Оценивают, как ПО работает при разных нагрузках
Виды:
- Нагрузочное тестирование - Позволяет определить максимально возможный пик, при котором система работает правильно
- Стресс-тестирование - Оценивает работу системы в нестандартных нагрузках

- Функциональное тестирование (Functional) - Проверяют соответствие ПО требованиям и задачам, для которых оно создано
Виды:
- Белый ящик - тестирование со знанием внутренней структуры системы
- Черный ящик - тестирование без знания внутренней реализации

- Дымовое тестирование (Smoke) - Тестируют основные функции программы, чтобы работали, как задумано, после внесения изменений в код
Пример: пользователь может успешно зайти на главную страницу магазина

### Правило пирамиды для тестирования

Стандарт тестирования в сфере информационных технологий. Правило является концепцией, которая описывает соотношение видов тестирования в разработке ПО

### Три модели для правила пирамиды

1. Стандартная пирамида тестирования. В этой модели большинство тестов составляют модульные тесты, которые находятся в нижней части пирамиды. На среднем уровне находятся интеграционные тесты. В верхней части пирамиды располагаются сквозные тесты. Большая часть тестов направлена на проверку мелких, изолированных компонентов программы
2. Перевернутая пирамида тестирования. В этой модели количество сквозных тестов превышает количество модульных и интеграционных тестов. Большинство усилий направлено на тестирование программы в ее целостности, а не на изолированные компоненты.
3. Песочные часы. В этой модели пирамиды наибольшее количество тестов - это модульные тесты. Далее идут сквозные тесты, между ними интеграционные тесты. Данный вид отображает баланс между тестирование компонентов и проверкой системы в целом



### Фреймворки для тестирования на С++

Фреймворк - набор инструментов и структур, с помощью которых можно создавать и запускать тесты, проверять отдельные части программы

Фреймворки работаю на макросах

Наиболее популярные - GTest/GMock, Catch, Mettle, Boost.Test

### Catch2

Современная среда модульного тестирования С++ с открытым исходным кодом. Библиотека Catch2 предоставляет простой и интуитивно понятный синтаксис для написания unit-тестов. Упрощает написание автоматических тестов

Режимы использования Catch2

1. Модульное тестирование  
2. Микро-бенчмаркинг - измерение производительности кода
3. Макросы в стиле BDD 

### Основные блоки библиотеки Catch2

Проверка истинности выражения:
`REQUIRE и CHECK - левая часть выражения == правая часть вражения`

Check - в случае ложности продолжит выполнение теста

Сравнение чисел с плавающей точкой 
Используется класс Approx

Методы Approx
- epsilon - для сравнения с допущением относительной ошибки
- margin - для сравнения с допущением абсолютной ошибки

Относительная:
```
Approx target = Approx(100).epsilon(0.01);
100.0 == target; // Очевидно выполнено
200.0 == target; // Очевидно не выполнено
100.5 == target; // Выполнено, так как допускаем отклонение в 1%
```

Абсолютная:
```
Approx target = Approx(100).margin(5);
100.0 == target; // Очевидно выполнено
200.0 == target; // Очевидно не выполнено
104.0 == target; // Выполнено, так как допускаем абсолютную ошибку 5
```

### TEST_CASE

Необходим для того, чтобы код работал корректно, разработчик тестирует разные варианты функциональности скрипта. Проверку в каждой такой функции в тестах упаковывают в отдельный блок - тест-кейс

Полезность:
1. Структурированные тесты.
2. Понятное описание, что тестируется.
3. Запуск тестов выборочно.

### Синтаксис TEST_CASE

`TEST_CASE( "название блока", "[название для фильтра]")'`

### SECTION

`SECTION( "название секции" );`

### Макрос INFO

Пример:
```
TEST_CASE("Тестирование факториала") {
	SECTION("Начало блока тестов факториала") {
		INFO("Тесты для факториала");
		// Ваши тесты здесь
	}
	
	SECTION("Дополнительная информация перед проверкой факториала 5") {
		INFO("Добавляем дополнительную информацию перед проверкой факториала 5);
		CHECK(Factorial(5) == 120);
	}
```



### Тестирование в Catch 2

Какую задачу решаем: Факториал числа n произведение всех чисел от 1 до n включительно. Например, факториал числа 5 равен 1 2 3 4 5, что равно 120

Тест для проверки истинности выражения
```
TEST_CASE("test factorial", "[factorial]") {
	CHECK(Factorial(0) == 1);
	
	SECTION("positive numbers") {
		CHECK(Factorial(1) == 1);
		CHECK(Factorial(5) == 120);
		CHECK(Factorial(10) == 3628800);
	}
}
```

Запуск тестов:
`Catch::Session().run(argc, argv);`,
для этого нужно подключить заголовочный файл: `#include<catch2/catch_session.hpp>`

Добавим макрос INFO:
`INFO("Factorial(5) calculated wrong");`
