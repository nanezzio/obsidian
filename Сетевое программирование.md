### Цели занятия

- Познакомиться с механизмом взаимодействия в сети (многоуровневый подход, интерфейсы и протоколы)
- Понять принципы создания и настройки сокетов Беркли
- Научиться создавать программы на языке С++ для обмена данными по сети при помощи сокетов Беркли


Сетевое программирование - это реализация обмена данными между определенными программными модулями, с использованием компьютерной сети.

Компьютерная сеть - это среда для связи, которую можно представлять как граф с узлами, являющимися субъектами взаимодействия, и ребрами, являющимися каналами для обмена данными между узлами


### Многоуровневый подход к передаче данных

Организация взаимодействия между устройствами сети является сложной задачей. Для решения используется универсальный прием - декомпозиция, то есть разбиение одной задачи на несколько задач-модулей


При декомпозиции часто используют многоуровневый подход:

- все множество модулей, решающих частные задачи, разбивают на группы и упорядочивают по уровням, образующим иерархию
- в соответствии с принципом иерархии для каждого промежуточного уровня можно указать непосредственно примыкающие к нему соседние вышележащий и нижележащий уровни
- группа модулей, составляющих каждый уровень, должна быть сформирована таким образом, чтобы все модули этой группы для выполнения своих задач обращались с запросами только к модулям соседнего нижележащего уровня
- с другой стороны, результаты всех модулей, отнесенных к некоторому уровню, могут быть переданы только модулям соседнего вышележащего уровня


Протокол - правила, определяющие последовательность и формат сообщений, которыми обмениваются сетевые компоненты, лежащие на одном уровне, но в разных узлах

Интерфейс - модули, реализующие протоколы соседних уровней и находящиеся в одном узле, взаимодействуют друг с другом в соответствии с четко определенными правилами с помощью стандартизированных форматов сообщений


### Модели OSI и DoD

Модель OSI (Open System Interconnection)
Разработана в 80х рядом международных организаций по стандартизации (ISO, ITU-T и другие). Модель сыграла значительную роль в развитии сетей.
Open System Interconnection (OSI) определяет различные уровни взаимодействия систем в сетях с коммутацией пакетов, дает им стандартные имена и указывает, какие функции должен выполнять каждый уровень.
Сетевое взаимодействие приложений всегда представлено стеком уровней

1. Сначала данные идут с верхних уровней на нижние и инкапсулируются в "обертки" протоколов, необходимые для сетевого обмена
2. Потом с нижних уровней - на верхние, при этом декапсулируются обратно в понятные уровню данные

Уровни модели OSI:
- Физический уровень (Physical Layer):
	- Передача битов: Отвечает за передачу необработанных битов по физической среде (кабели, оптоволокно, радиоволны)
	- Определение характеристик: Устанавливает электрические, механические и функциональные характеристики оборудования
	- Кодирование сигналов: преобразует цифровые данные в сигналы для передачи по физическим носителям (например, Ethernet)
- Канальный уровень (Data Link Layer)
	- Формирование кадров: Упаковывает данные в кадры, добавляя служебную информацию (например, адреса отправителя и получателя)
	- Контроль ошибок: Обеспечивает обнаружение и исправление ошибок при передаче данных
	- Управление доступом к среде: Регулирует доступ к физической среде с использованием MAC-адресов
- Сетевой уровень (Network Layer)
	- Маршрутизация: Определяет оптимальные пути для передачи пакетов данных между различными сетями
	- Логическая адресация: Использует IP-адреса для идентификации узлов в сети
	- Фрагментация пакетов: Разделяет большие пакеты на более мелкие для передачи через сети с ограниченной пропускной способностью
- Транспортный уровень (Transport Layer)
	- Доставка данных: Обеспечивает надежную передачу данных между приложениями, используя протоколы TCP и UDP
	- Контроль целостности: Отслеживает потери пакетов и обеспечивает их повторную передачу при необходимости
	- Управление потоком: Регулирует скорость передачи данных для предотвращения перегрузки сети
- Сеансовый уровень (Session Layer)
	- Управление сессиями: Отвечает за установку, управление и завершение сеанса связи между приложениями
	- Синхронизация: Обеспечивает синхронизацию задач и обмен данными между приложениями
- Уровень представления (Presentation Layer)
	- Преобразование форматов: Кодирует и декодирует данные, а также выполняет их сжатие и шифрование
	- Подготовка данных к передаче: Приводит данные к формату, понятному приложению
- Прикладной уровень (Application Layer)
	- Интерфейс для приложений: Предоставляет сетевые услуги приложениям, позволяя им взаимодействовать с сетью
	- Протоколы взаимодействия: Включает протоколы для работы с вебом (HTTP), электронной почтой (SMTP), передачей файлов (FTP) и другими сетевыми сервисами


### Модель DoD (Department of Defence)

Это сетевая модель, разработанная Министерством обороны США, которая описывает структуру передачи данных в компьютерных сетях.
Была разработана в 1970-х годах для обеспечения надежной связи в условиях военных действий, включая ядерные конфликты. Это требование привело к созданию устойчивой сети, способной функционировать даже при повреждении отдельных узлов
Она также известна как модель TCP/IP и служит основой для протоколов, используемых в Интернете

### Структура уровней DoD

- Уровень приложений (Process/Aplication): Соответствует верхним трем уровням модели OSI (приложение, представление, сессия).
	- Обработка данных пользователей: Этот уровень включает протоколы, которые обрабатывают данные используемые конечными пользователями, такие как Teinet (удаленный доступ), FRP (передача файлов) и SMTP (доставка электронной почты)
	- Управление обменом данными: Обеспечивает управление передачей данных между приложениями с помощью протоколов, таких как SNMP и DNS, которые помогают сопоставлять имена адресам и управлять устройствами
- Транспортный уровень (Transport): Соответствует транспортному уровню модели OSI
	- Контроль целостности данных: Отвечает за надежную передачу данных между узлами, включая механизмы для обнаружения и исправления ошибок
	- Установление соединений: Управляет установкой и завершением соединений между отправителем и получателем, обеспечивая необходимый уровень надежности передачи данных
- Межсетевой уровень (Internet): Соответствует сетевому уровню модели OSI
	- Маршрутизация данных: Обеспечивает маршрутизацию пакетов данных между различными сетями, используя IP-протоколы
	- Логическая адресация: Отвечает за назначение IP-адресов устройствам в сети и фрагментацию пакетов для передачи
- Уровень сетевого доступа (Networl Access): Соответствует канальному и физическому уровням модели OSI. Он содержит протоколы, которые обеспечивают физическую передачу данных между устройствами сети
	- Физическая передача данных: Обеспечивает передачу данных по физической среде, включая спецификации для различных типов сетевых интерфейсов
	- Адресация устройств: Устанавливает физические адреса для устройств в сети и управляет доступом к среде передачи

### Протоколы транспортного уровня

1. UDP (User Datagram Protocol) - Транспортный протокол без установления соединения
2. TCP (Transmission Control Protocol) - Специально разработан для обеспечения надежной передачи данных по не надежной сети

### UDP
- Отвечает только за отсылку пакетов
- Не занимается:
	- контролем потока
	- контролем ошибок
	- повторной передачей после приема испорченного сегмента
- UDP пакет состоит из 8-байтного заголовка, за которым следует поле полезной нагрузки

### UDP пакет
- Состоит из 8-байтного заголовка, за которым следует поле полезной нагрузки
- Два номера портов служат для идентификации конечных точек внутри отправляющей и принимающей машин
- Поле Длина UDP содержит информацию о длине сегмента, включая заголовок и полезную нагрузку


### TCP
- Ключевое свойство TCP, которое определяет всю структуру протокола - у каждого байта есть свой 32-разрядный порядковый номер
- Отправляющая и принимающая TCP-сущности обмениваются данными в виде сегментов
- Сегмент состоит из фиксированного 20-байтового заголовка, за которой могут следовать байты данных

### Протокол скользящего окна
Основной протокол, используемый TCP-сущностями

- При передаче сегмента отправитель включает таймер
- Когда сегмент прибывает в пункт назначения, принимающая TCP-сущность посылает обратно сегмент с номером подтверждения, равным порядковому номеру следующего ожидаемого сегмента
- Если время ожидания подтверждения истекает, отправитель посылает сегмент еще раз


### IP адресация (Internet Protocol)

Протокол межсетевого взаимодействия IP:
- Является основой транспортных средств стека протоколов TCP/IP
- Обеспечивает передачу пакетов от отправителя к получателям через объединенную систему компьютерных сетей
- Основой транспортной системы является IP адресация
- IP-адреса представляют собой основной тип адресов, на основании которых сетевой уровень передает пакеты между сетями
- На сегодняшний день используются две версии протокола IP; четвертая и шестая

### Версии протокола IP

- IPv4 - Записывается в виде четырех чисел, представляющих значения каждого байта в десятичной форме и разделенных точками. Пример: 109.26.17.100
- IPv6 - Записывается в виде восьми групп шестнадцатеричных чисел, разделенных двоеточиями. Пример: 2001:0db8:85a3:0000:0000:8a2e:0370:7334

### Изменение IPv6 по отношению к IPv4

- Расширение адресации
	В IPv6 длина адреса расширена с 32 до 128 бит, что позволяет обеспечить больше уровней иерархии адресации, увеличить число адресуемых узлов, упростить авто-конфигурацию
- Спецификация формата заголовков
	Некоторые поля заголовка IPv4 отбрасываются или делаются опционными, уменьшая издержки, связанные с обработкой заголовков пакетов с тем, чтобы уменьшить влияние расширения длины адресов в IPv6
- Улучшенная поддержка расширений и опций
- Возможность пометки потоков данных
	Введена возможность помечать пакеты, принадлежащие определенным транспортным потокам, для которых отправитель запросил определенную процедуру обработки
- Идентификация и защита частных обменов
	В IPv6 введена спецификация идентификации сетевых объектов или субъектов, для обеспечения целостности данных и при желании защиты частной информации

### Сокеты Беркли

Сокет (Узел) - представляют собой интерфейс программирования приложений (API). Технология сокетов была разработана в университете Беркли, затем IEEE стандартизировали их в рамках POSIZ (Portable Operating System Interface)

Сокеты - одна из фундаментальных технологий, на которой основан интернет. Все современные операционные системы так или иначе реализуют интерфейс POSIX-сокетов

Программирование сокетов - это способ соединения двух узлов в сети для связи друг с другом. Один сокет (узел) прослушивает определенный порт на IP, в то время как другой сокет обращается к другому для формирования соединения

Сокетное API уровня транспортного протокола и выше обычно предполагает наличие:
1. Сервера - тот, кто предоставляет услугу. В ответ на запрос он может вернуть данные или выполнить некие действия. Важно, что сервер ожидает запроса
2. Клиента - тот, кто запрашивает услугу. Как правило, в ответ он получает данные


### Этапы создания сервера

- Создание сокета
- Настройка сокеты
- Соединение сокета с локальным адресом 
- Прослушивание сокета
- Подключение клиента

### Создание сокета

`int sockfd = socket(domain, type, protocol)`

- sockfd - дескриптор сокета, целое число (по аналогии с дескриптором файла)
- doamin - целое число, указывает коммуникационный домен или семейство протоколов. Существуют предопределенные константы задающие нужное семейство:
	- AF_LOCAL - для связи между процессами на одном хосте
	- AF_INET - для связи между процессами на разных хостах, по протоколу IPv4
	- AF_INET6 - для связи между процессами на разных хостах, по протоколу IPv6
	- AF_BLUETOOTH - для связи между процессами на разных хостах, по беспроводному протоколу bluetooth
- type - тип транспортного протокола: SOCK_STREAM: TCP, SOCK_DGRAM: UDP
- protocol - тип сетевого протокола, для IP равно 0. Это то же число, которое отображается в поле протокола в заголовке IP-пакета

### Настройка сокета

`int setsockop(int sockfd, int level, int optname, const void* optval, socklen_t optlen);`

Задает параметры сокета, на который ссылается файловый дескриптор sockfd
Это совершенно необязательно, но помогает в повторном использовании адреса и порта. Предотвращает, например, ошибки повторного использования адреса

### Соединение сокета с локальным адресом

`int bind(int sockfd, const struct sockaddr* addr, socklen_t addrlen);`

- sockfd - дескриптор сокета
- addr - указатель на структуру адреса
- addrlen - размер структуры адреса
возвращаемое значение - 0 при успехе либо - 1 при неудаче

Структура адреса sockaddr хранит такие параметры как тип протокола, IP адрес и номер порта. За исключением параметра sin_family, содержимое sockaddr использует специальный порядок байт

```
struct sockaddr_in 
{
	short sin_family;
	u_short sin_port;
	struct in_addr sin_addr;
	char sin_zero[8];
}
```

### Прослушивание сокета

`int listen(int sockfd, int backlog);`

- sockfd - дескриптор сокета
- backlog - размер очереди подключений
возвращаемое значение - 0 при успехе либо - 1 при неудаче

Функция переводит сокет сервера в пассивный режим, где он ждет, когда к серверу придет запрос от клиента для установления соединения. Параметр backlog определяет максимальную длину, до которой может увеличиться очередь ожидающих подключений для sockfd. Если запрос на подключение поступит, когда очередь заполнена, клиент может получить ошибку с указанием ECONNREFUSED

### Подключение клиента

`int new_socket
